# 投递跟踪

用于岗位管线、人脉沟通、跟进和 Offer 评估。

## 反馈回流原则

跟踪表不是只记录“投了多少”。每次无反馈、已读不回、HR 追问、面试失败、薪资卡住或用户状态变差，都要回流到下一轮动作。

默认分类：

```text
岗位匹配问题：岗位要求和候选人证据距离过大，或岗位真实方向判断错误。
材料呈现问题：关键词、结构、作品、数字、证据或附件没有回答 JD。
面试表达问题：经历能写但说不清，项目细节、边界或迁移逻辑不稳。
渠道时机问题：平台、投递时间、岗位热度、HC 变化、重复投递或联系人不对。
用户状态问题：焦虑、疲惫、回避、羞耻感过强，导致动作失控或停摆。
```

复盘动作：

- 同一类岗位连续无反馈：先改在线关键词、附件简历标题、打招呼钩子和岗位池，不直接加倍投递。
- 同一缺口连续被 HR 或面试官追问：把它转成补证据、模拟作品、学习附录、项目深挖或面试故事任务。
- 同一公司多渠道冲突：先查投递记录和联系人，再决定是否跟进或暂停。
- 用户明显低落：先缩小为 1 个最小行动，必要时休息，不用海投缓解焦虑。

## 投递跟踪表

```markdown
| 公司 | 岗位 | 投递渠道 | URL | 状态 | 匹配度 | 简历版本 | 联系人 | 下一步 | 日期 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
```

状态建议：

```text
researching -> tailored -> ready -> authorized -> applied -> stopped -> follow-up -> interview -> offer -> rejected -> archived
```

结构化投递队列使用 `references/application-copilot.md` 的 `job_card` 作为岗位主记录。跟踪表只记录用户能看懂的摘要；需要机器可读或批量复盘时，追加投递事件。

```yaml
application_event:
  event_id:
  job_id:
  batch_id:
  event_type: collected/screened/tailored/ready/authorized/sent/sent_by_agent/stopped/stopped_by_guardrail/followed_up/replied/interview/rejected/offer/archived
  timestamp:
  channel:
  actor: user/agent/system
  notes:
  feedback:
  next_action:
```

规则：

- `job_id` 必须能回到具体岗位卡、岗位链接或投递批次。
- `ready` 表示投递包已准备但未对外发送；`authorized` 表示用户确认了单岗位或批次授权卡；`sent` 表示用户本人发送；`sent_by_agent` 表示 Agent 在授权范围内完成发送；`stopped_by_guardrail` 表示触发验证码、风控、公司不明、材料缺失、敏感字段或授权不完整等停止条件。
- 同公司、同岗位名、同 URL、同岗位编号或疑似猎头重复岗位，先标重复风险，再决定是否投递。
- 有 HR 回复、面试追问或无反馈复盘时，把反馈写回 `feedback` 和 `next_action`，不要只改简历正文。

## 人脉私信

```text
你好 [Name]，我正在了解 [role/domain] 方向，看到你在 [company/team] 的经历。
我特别想了解 [specific reason]。
如果方便的话，能否请教你 1 个问题：这个岗位实际最看重什么能力？
```

保持低压力。不要一上来就让陌生人帮忙找工作。

## 跟进

这些情况可以跟进：

- 招聘方在合理时间内没有回复。
- 新证据能增强申请。
- 面试后需要发送感谢和补充信息。

跟进不只问“有结果了吗”。优先提供增量信息，例如补充作品样例、更新简历版本、确认可到岗时间、补充面试中未说清的问题。没有增量信息时，最多轻跟进 1-2 次，之后归档或进入下一批岗位。

## 投递去重

同公司同岗位不要在多个渠道乱投。跟踪表里记录：

```text
公司、岗位名、岗位编号、渠道、简历版本、推荐人/猎头、投递日期
```

如果已经通过内推或猎头进入流程，再从官网或 BOSS 重复投递前，先确认是否会冲突。

## 面试复盘模板

面完 30 分钟内记录：

```markdown
| 项目 | 事实记录 | 情绪标签 | 下一步 |
| --- | --- | --- | --- |
| 答得好的 3 点 |  | 自信/平静/意外 | 保留并强化 |
| 答得不好的 3 点 |  | 紧张/混乱/羞耻 | 补材料或补回答 |
| 新出现的问题 |  | 意外/焦虑 | 加入问题库 |
| 面试官透露的信息 |  | 希望/失望/不确定 | 调整公司理解 |
| 下一轮要补的材料 |  | 有把握/没把握 | 明确补证据动作 |
| 我的整体状态 |  | 亢奋/正常/低落 | 判断是否需要休息 |
```

## 面试探行业与作品迭代

面试不只用于争取录用，也可以用于了解行业真实要求。一次失败面试如果能带回判断标准、常见追问和材料短板，就不是纯损耗。

复盘时追加：

```markdown
| 复盘项 | 记录 | 下一步动作 |
| --- | --- | --- |
| 面试官真正关心的问题 |  | 加入 JD 题目库 |
| 被追问最深的经历/作品 |  | 强化证据、补截图、补数字 |
| 对方不买账的表达 |  | 改写简历 bullet 或面试说法 |
| 新了解到的岗位标准 |  | 更新目标岗位要求地图 |
| 可以转成作品集的反馈 |  | 替换低分作品页或新增案例页 |
```

迭代规则：

- 不把一次拒绝直接解释成“我不行”；先判断是岗位不匹配、证据不够、表达不清，还是作品没有回答 JD。
- 如果连续 3 次被问到同一缺口，必须转成学习附录、模拟作品或简历重写任务。
- 如果面试官对某个作品提出有价值反馈，优先更新作品集，而不是只改自我介绍。
- 不用攻击性“踢馆”表达。可以做前置思考、改进提案或模拟方案，但对外说法要克制、请教式、尊重对方已有工作。

## 投递反馈复盘卡

当对话里不适合展开完整表格时，使用这个短卡：

```markdown
## 这轮反馈
- 发生了什么：
- 最可能属于哪类问题：
- 先改 1-3 个变量：
- 下一轮测试岗位：
```

使用规则：

- 不把一次拒绝解释成“用户不行”。
- 不因为一次失败推翻全部方向。
- 不在没有反馈时无限重写简历；先判断是否缺关键词、缺证据、缺渠道或岗位池过窄。
- 不把面试官的所有意见都当真理；只吸收可验证、可复用、和目标岗位共性相关的反馈。

## 情绪复盘衔接

如果用户出现投递无反馈、连续面试失败、自我怀疑、空窗期焦虑或明显求职疲惫，同时读取 `emotional-resilience.md`。

复盘时区分两类问题：

```text
工具型问题：简历、话术、项目证据、岗位池、面试回答。
状态型问题：焦虑、羞耻、自我怀疑、能量过低、持续回避。
```

如果连续多次复盘都出现“低落/羞耻/想放弃”，优先建议短暂停止投递，恢复节奏后再做策略调整。

## 背调准备

背调可能核验：

- 学历、工作时间、岗位名称。
- 离职原因、绩效或工作表现。
- 犯罪记录、商业冲突、竞业限制。
- 薪资证明或流水，视公司要求而定。

准备：

- 确认简历时间线和社保/离职证明大体一致。
- 提前准备学历证明、离职证明、推荐人信息。
- 有空窗、短经历、岗位名称不一致时，提前准备解释口径。
- 薪资流水涉及隐私，优先确认用途、接收主体和是否可脱敏。

## Offer 评估

比较：

```text
岗位匹配
直属上级/团队
学习曲线
薪酬
稳定性
地点/远程
成长路径
风险
```

接受前应拿到书面条款。不要只凭口头 offer 做最终决定。

口头 Offer 也有时效。拿到后确认：

```text
书面 Offer 预计何时发？
回复截止日期是什么时候？
入职日期是否可协商？
未确认薪资/试用期/社保公积金前，不要急着口头承诺。
```

## 劳动关系主体检查

国内 Offer 不能只看岗位名和薪资，还要确认“和谁签、谁发薪、在哪工作、归谁管理”。

```markdown
| 项目 | 要确认什么 | 风险 |
| --- | --- | --- |
| 签约主体 | 劳动合同和哪家公司签 | 招聘品牌与签约主体不一致 |
| 发薪主体 | 工资由谁发放，个税由谁申报 | 第三方发薪、薪资证明不一致 |
| 社保主体 | 社保/公积金在哪个城市、由谁缴纳 | 到手、落户、购房、医保受影响 |
| 工作地点 | 自有办公、客户现场、门店、驻场 | 实际通勤和工作内容漂移 |
| 管理关系 | 直属上级属于哪家公司/部门 | 外包或派遣岗位成长受限 |
| 合同类型 | 劳动合同、劳务合同、实习协议、派遣、外包 | 权益和解除规则不同 |
```

如果出现“外包、派遣、驻场、第三方、人力资源公司代签、项目合同、劳务合作”等词，不直接否定，但要把岗位稳定性、社保、转正机会、绩效归属和简历呈现口径问清楚。

## 背调授权边界

背调前先确认：

```text
背调公司/接收主体是谁？
背调范围包括哪些内容？
是否需要本人书面授权？
是否会联系现任/前任公司？联系前是否会告知？
薪资流水、社保记录、离职证明是否可脱敏？
```

未获得明确授权和流程说明前，不建议主动发送薪资流水、完整身份证件、完整家庭信息或无关隐私材料。

## 谈薪与 Offer 确认清单

```markdown
| 项目 | 要确认什么 | 风险 |
| --- | --- | --- |
| base | 月薪、发薪月数、试用期是否折扣 | 口头月薪和 offer 不一致 |
| 绩效/提成 | 计算方式、发放周期、达成条件 | 只说“综合薪资”不说结构 |
| 年终奖 | 几薪、是否保底、入职未满一年怎么算 | 写法模糊 |
| 社保公积金 | 缴纳基数、比例、城市 | 按最低基数缴纳 |
| 试用期 | 时长、工资、转正标准 | 试用期过长或标准不清 |
| 工作地点 | 城市、办公点、驻场/出差 | 实际地点变化 |
| 工作时间 | 排班、加班、调休、远程 | 只说“弹性”不说边界 |
| 汇报关系 | 直属上级、团队规模、岗位归属 | 岗位职责漂移 |
| 背调 | 时间、范围、授权材料 | 未提前解释空窗/离职 |
| 竞业/保密 | 是否签署、范围、补偿 | 约束过宽或无补偿 |
| 三方/网签 | 签约主体、违约条款、学校流程 | 应届生毁约成本 |
```

谈薪口径：

```text
我对这个岗位是有兴趣的。结合岗位要求、所在城市和我的经历，我期望整体薪酬在 X-Y 区间。也想进一步了解贵司薪酬结构，包括 base、绩效/年终、试用期和社保公积金口径。
```

接受 offer 前输出：

```text
已确认条款
待确认条款
红旗风险
是否建议接受
下一步沟通话术
```
