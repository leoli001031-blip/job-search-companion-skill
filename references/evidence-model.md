# 证据模型

用于判断哪些内容可以安全进入简历、作品集和面试回答。

证据整理不是为了做一个笨重档案库，而是把用户零散材料拆成后续能复用的经历卡片。每张卡只保留求职有用的信息。

## 经历卡片

```markdown
## 经历 / 项目 / 活动名称

- 来源材料：
- 时间与场景：
- 候选人角色：
- 具体动作：
- 交付物或结果：
- 可量化结果：
- 当前证据：
- 可迁移能力：
- 可用于匹配的岗位/场景：
- 还缺什么：
```

同一段经历可以服务多个方向。不要在这一步急着写成最终简历 bullet，先保留事实和证据。

## 证据等级

| 等级 | 定义 | 例子 | 简历使用建议 |
| --- | --- | --- | --- |
| 强证据 | 第三方、公开、可渲染、可直接检查的证明 | 实习证明、获奖证书、成绩单、公开链接、作品、上线页面、PDF、Demo 视频、截图、测试日志、公开文章、GitHub 仓库 | 可作为核心事实，但仍要判断是否适合公开披露 |
| 中证据 | 候选人拥有、能现场解释或展示的材料 | 本地文档、私有原型、内部笔记、活动照片、未公开截图、课程作业 | 用克制表达，优先脱敏后做作品集或面试展示 |
| 弱证据 | 只有口头描述，没有材料支撑 | “我提升了 X”“我参与过 Y”“老师/同事评价不错” | 可以做回忆线索，写入简历前必须追问确认或降级表达 |
| 不安全 | 无法验证，或超过已知证据 | 夸大的指标、借来的案例、无法证明的岗位/奖项/成果 | 不使用 |

证据等级只回答“这件事能不能支撑候选人的表述”，不等于“可以直接公开”。例如薪资流水、内部表格、客户名单、未公开运营数据，即使能证明经历，也不应进入投递版简历。

## 披露安全等级

| 等级 | 适合放在哪里 | 处理方式 |
| --- | --- | --- |
| 公开安全 | 简历、作品集、面试展示均可使用 | 使用公开链接、公开作品、公开证书或已发布内容 |
| 脱敏后可用 | 作品集、案例页、面试展示 | 隐去公司敏感数据、客户/人员姓名、内部系统、金额明细和账号信息 |
| 仅内部参考 | 素材整理和面试准备 | 不放进简历和作品集，只用于帮助候选人回忆经历、准备口径 |
| 不使用 | 任何对外材料都不使用 | 涉及隐私、违法风险、保密协议、他人作品或无法确认真实性 |

简历正文只写可读摘要。需要证明时，优先额外交付：

```text
脱敏作品集 / 案例页 / 证据附件 / 面试展示页
```

不要为了“证据充分”把证明材料本身塞进简历正文。

## 内容安全检查

加入任何重要表述前，检查：

- 谁做了这件事？
- 具体组织、交付、服务、教授、销售、分析、设计、搭建或改善了什么？
- 用了什么工具、方法或流程？
- 有什么证据？
- 数字是否确认？
- 候选人能否经得起面试追问？

## 合理包装边界

可以做：

- 把真实经历翻译成目标岗位语言。
- 调整经历排序，把更相关的证据放前面。
- 强调真实角色中和 JD 相关的部分。
- 用更准确的业务/能力词描述原本朴素的任务。
- 对短期能补的材料，建议补截图、复盘、案例页、作品页或说明文档。

动作词降级：

```text
角色边界不明：协助、参与、支持、整理、跟进、配合、使用过。
确认模块责任：负责某个模块、维护某张表、跟进某类事项。
确认主要责任：主导、独立完成、搭建、推进、优化。
```

用户急投时先用更低一档动作词写出可读材料，把需要确认的强动作词放入复核清单。

成果词也要降级：没有数字、截图、材料或用户确认时，不把“提升、转化、增长、优化、闭环、提高效率、降低成本”写成确定结果；改写成“支持/辅助/用于/为后续提供参考”。

不能做：

- 虚构公司、岗位、项目、奖项、证书或结果。
- 把旁观、协助、参与写成主导或负责。
- 把学习中、了解过写成熟练掌握或独立交付。
- 把无法确认的数字写成确定成果。
- 使用面试中无法解释清楚的关键词。

包装后的表述必须能回答：

```text
这件事真实发生了吗？
候选人具体做了哪一部分？
为什么可以对应这个 JD 要求？
有没有材料能证明，或至少能现场解释？
如果面试官追问细节，候选人能讲清楚吗？
```

## Bullet 公式

优先使用：

```text
动作 + 对象 + 方法/场景 + 结果/证据
```

示例数字只用于展示结构，不能迁移到候选人材料。用户没有确认数字时，用 `[待补充]`、`X` 或定性表达；用户没有确认角色边界时，用“参与/协助/支持”等更保守动作词。

结构示例：

```text
协助学院迎新活动志愿者排班与现场协调，支持 [待补充] 名志愿者完成 [待补充] 名新生报到引导，并沉淀现场问题清单作为第二年流程优化参考。
```

避免：

```text
参与活动，认真负责，沟通能力强。
```

## 成果量化补采

生成简历或项目案例时，如果用户没有提供数字，必须单独列出可补数字。不要因为缺数字就停下，也不要为了显得好看而编数字。

可复制的 bullet、话术和案例页中，不要用真实感数字或确定性成果做占位，例如 `30+`、`200+`、`下降约 X%`、`零差错`、`熟练掌握`、`主导`、`提升效率`。除非这些内容来自用户确认或可验证材料，否则写成 `[待补充]`、`待确认`、定性支持作用，或放入量化缺口表。禁用清单是内部检查项，面向用户解释时优先概括为“未确认数字/强动作词/成果词”，不要把清单原样写进可投递成品。

常见量化维度：

```text
规模：人数、客户数、商户数、活动数、项目数、门店数、素材数、视频时长。
效率：从多久缩短到多久、减少多少步骤、减少多少修改轮次、节省多少人工时间。
产出：交付多少张图、多少条视频、多少页报告、多少版方案、多少个模板。
质量：采纳率、通过率、满意度、返工率、错误率、复用次数。
业务：销售额、转化率、客单价、复购、线索数、到店量、报名人数。
学习/培训：培训人数、场次、完成率、独立使用比例、反馈评价。
技术/项目：支持多少输入类型、处理多少样例、覆盖多少流程节点、Demo 是否可演示。
```

一线、服务、蓝领、门店、销售、教育、医疗、行政、家政、客服等岗位也可以量化，不限项目制工作：

```text
服务量：接待人数、订单数、班次数、工单数、巡检次数、上课节数、护理/服务对象数。
稳定性：出勤、准点率、投诉率、返工率、安全事故、差错率、续约/复购。
执行质量：标准动作完成率、陈列/卫生/台账检查、活动落地数量、培训通过率。
协作对象：门店、班级、客户、供应商、部门、社区、病区或现场团队数量。
```

量化缺口表：

```markdown
| 经历/项目 | 当前可写成果 | 缺少的数字 | 怎么补 | 补完后的简历写法 |
| --- | --- | --- | --- | --- |
```

如果无法量化，可以退而求其次使用：

```text
具体场景 + 明确交付物 + 可展示证据 + 面试可解释结果
```

