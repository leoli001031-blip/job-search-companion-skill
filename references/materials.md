# 求职材料

用于简历、求职信、作品集和项目案例。

## 简历结构选项

标准候选人：

```text
联系方式
个人总结
技能
工作经历
项目/实习/校园经历
教育背景
证书 / 获奖 / 作品集
```

学生/应届/实习候选人：

```text
联系方式
求职意向 / 个人总结
教育背景
实习经历
项目/课程/竞赛经历
校园/社团/志愿经历
获奖 / 证书 / 语言
技能 / 作品链接
```

转型或项目驱动型候选人：

```text
联系方式
定制个人总结
核心技能
相关项目 / 案例
可迁移经历
教育 / 证书
链接
```

国内传统投递 / 平台友好版：

```text
联系方式
求职意向
教育背景
个人优势 / 岗位匹配
专业技能
项目 / 实习 / 工作经历
校园 / 社团 / 证书 / 作品链接
```

使用条件：

- JD 明确写学历、专业、证书或应届/校招要求时，教育背景不要压到最后。
- 候选人教育经历能覆盖硬门槛或缓解疑虑时，教育背景可以放在个人优势之后或之前。
- 转型候选人如果项目证据强，但学历/专业有疑虑，可以采用“教育前置 + 项目前置”的折中结构：先让 HR 看到学历线，再用项目证明岗位能力。
- 传统行业、国企/央企/校招/实习/管培场景更重视教育、时间线和证书；互联网作品型岗位可以更突出项目和作品。

## 生成前重点确认

生成完整简历正文或 PDF 前，先做一次“编排建议与重点确认”。目标不是让 AI 自作主张，而是让用户确认这版简历最想突出什么。

触发条件：

- 用户要求“写一份简历”“改一版简历”“导出 PDF 简历”“针对 JD 定制简历”。
- 用户素材里存在多个可能重点，例如教育、实习、项目、作品、工作经历、校园获奖或证书都能写。
- 用户有多段经历或模板容量有限，必须决定哪些放进简历正文、哪些压缩、哪些移入作品集或案例页。
- 用户没有明确说“这版重点突出项目/实习/教育/作品”。

快速出稿例外：

- 如果用户明确要求“直接快速出一版草稿”，且没有同时要求 PDF/终稿/定稿，也没有明确说“还没确认重点”，可以先出草稿。
- 快速草稿必须标注当前编排假设、待确认信息、不能硬写的数字、成果量化补采表和评分表。
- 如果用户同时说“还没确认重点/信息不足/要 PDF/要终稿”和“直接写”，优先执行确认门：先输出编排方案、风险、AI 推荐和最小确认问题。可以附“非终稿草稿骨架”或 bullet 池帮助急投，但不生成完整定稿或 PDF。

输出格式：

```markdown
## 简历编排建议与重点确认

基于你提供的材料，我建议这版简历可以有 2-3 种编排。每个方案必须给完整 section 顺序，不要只说“项目/教育/经历前置”：

| 方案 | 重点 | 完整 section 顺序 | 适合场景 | 风险 |
| --- | --- | --- | --- | --- |
| A | 项目 / 作品前置 | 基本信息 -> 求职意向 -> 定制总结 -> 核心项目/作品 -> 相关经历 -> 技能 -> 教育/证书 -> 链接 | 目标岗位重视项目证据，正式经历不完全同线 | 需要作品链接或可追问细节支撑 |
| B | 工作 / 实习经历前置 | 基本信息 -> 求职意向 -> 定制总结 -> 工作/实习经历 -> 项目/作品 -> 技能 -> 教育/证书 -> 链接 | 相关公司、岗位、业务场景更能打动 HR | 项目亮点可能被压后 |
| C | 教育 / 证书前置 | 基本信息 -> 求职意向 -> 教育背景 -> 证书/获奖 -> 实习/项目/工作经历 -> 校园/社团 -> 技能 -> 链接 | 应届、实习、校招、国企或 JD 卡学历/专业/证书 | 如果后续经历弱，容易显得单薄 |

AI 推荐：[方案 X]，原因：[基于素材和 JD 的 2-3 条理由]。
请你确认：这版简历优先突出 A / B / C，还是你有其他重点？

## 经历取舍清单

| 素材/经历 | 建议处理 | 原因 | 需要用户确认 |
| --- | --- | --- | --- |
| [经历 1] | 放入简历正文 | [最贴近 JD / 证据强 / 能解释转型] | [是/否] |
| [经历 2] | 压缩为 1 行 | [辅助证明，不宜占太多篇幅] | [是/否] |
| [经历 3] | 移入作品集/案例页 | [材料丰富但简历放不下] | [是/否] |
| [经历 4] | 暂不使用 | [关系弱或解释成本高] | [是/否] |
```

常见方案：

| 方案 | 完整 section 顺序 | 适合情况 |
| --- | --- | --- |
| 项目 / 作品能力 | 基本信息 -> 求职意向 -> 定制总结 -> 核心项目/作品 -> 项目成效/证据 -> 相关经历 -> 技能 -> 教育/证书 -> 链接 | 作品、项目、Demo、GitHub 能直接证明岗位能力 |
| 工作 / 实习经历 | 基本信息 -> 求职意向 -> 定制总结 -> 工作/实习经历 -> 关键成果 -> 项目/作品 -> 技能 -> 教育/证书 -> 链接 | 公司、岗位、实习或工作场景和 JD 高度相关 |
| 教育 / 证书 / 校园 | 基本信息 -> 求职意向 -> 教育背景 -> 证书/获奖/语言 -> 实习/项目 -> 校园/社团/志愿 -> 技能 -> 作品链接 | 应届、实习、校招、国企、传统行业或 JD 卡学历/证书 |
| 综合稳妥版 | 基本信息 -> 求职意向 -> 教育背景 -> 个人优势/岗位匹配 -> 专业技能 -> 经历/项目 -> 证书/作品链接 | 国内传统投递、用户不确定重点或需要兼顾平台解析 |
| 转型折中版 | 基本信息 -> 求职意向 -> 教育背景 -> 定制总结 -> 相关项目/作品 -> 可迁移经历 -> 技能 -> 证书/链接 | 转型候选人需要先过学历/稳定性，再用项目证明目标能力 |

输出编排方案时，还要说明“为什么这样排”：

```text
顶部解决什么信任问题：
中段证明什么岗位能力：
后段补充什么可信度或检索关键词：
哪些素材不进正文，改放作品集/案例页：
```

确认后再生成简历正文。不要把“AI 推荐”当成最终决定；最终围绕用户选择编排。

## 输出状态

简历相关产物用短句标清成熟度，不需要复杂表格：

```text
本轮状态：草稿骨架，需确认重点。
本轮状态：可复制草稿，未生成文件。
本轮状态：可投递版，关键事实已确认。
本轮状态：PDF 排版方案，未导出 PDF。
本轮状态：已生成文件，路径已验证。
```

如果没有实际写入并验证文件，不要使用“已生成文件”状态。

## 简历规则

- 招聘系统解析版保持单栏、文本可读、section 标准。
- 联系方式放正文，不放页眉/页脚。
- 招聘系统解析版文件避免表格、技能条、文本框、扫描图片 PDF 和装饰图形。
- 视觉 PDF 要克制，设计服务于阅读，不掩盖内容薄弱。
- 链接只在能证明某件事时加入。
- 不要直接把所有素材塞进固定模板。先判断候选人可能的叙事重点，并让用户确认本版简历优先突出项目、经历、教育、证书、作品还是综合稳妥。
- 优先使用具体经历要点，而不是宽泛自我评价；经历可以来自工作、实习、校园、竞赛、项目、志愿或作品。
- 每版草稿都要附缺口清单和成果量化补采表。
- 每版草稿都要附简历评分表和使用说明；评分维度见 `output-evaluation.md`。
- 简历中的重要经历优先写成“动作 + 场景 + 交付物 + 量化结果/证据”；如果用户还没有提供量化结果，先用可信的交付物和证据写一版，再明确提示用户补哪些数字后可以替换。
- 技能区不要只堆工具名。按岗位能力分组：岗位核心技能、工具/方法、业务场景、沟通协作、语言/证书；每组控制在 4-8 个高相关词。
- 对国内平台，技能区要兼顾关键词命中和真实可追问：能解释的写“实践”，只了解的写“了解/使用过”，不要写“精通”。
- 简历正文不要写解释性防御句，例如“虽然不是相关背景”“非科班/非算法/非技术背景”“只是协助”“没有完整负责”。这些内容放到内部风险提示或面试口径里；正文只写真实任务、交付物、协作对象和岗位相关性。
- 如果必须说明边界，用动作词强度体现，例如“协助/参与/支持/整理/维护”，不要在 bullet 里反复解释“我没有/我不是/我不会”。
- 原型、自测、模拟任务或脱敏样例可以写进简历和作品页，但用准确限定词内置边界即可；不要在正文后反复追加“仍需真实业务验证”“尚未上线”“没有真实用户试用”等拆台句。这些放到内部使用说明或面试追问口径。

## 国内平台简历规则

面向国内平台时，同时准备：

```text
平台在线简历：用于搜索、初筛、系统匹配和平台沟通。
PDF 附件简历：用于下载、转发、业务负责人阅读和面试归档。
一句话优势：用于 BOSS/猎聘/内推私信开场。
作品集/案例页：用于设计、内容、产品、运营、项目型候选人。
```

要求：

- 在线简历和附件简历的时间线、公司/学校、岗位名、项目名一致。
- 附件文件名优先包含姓名、岗位方向和城市；电话后四位只在用户愿意或渠道确实需要检索时加入，不默认暴露完整手机号。
- 多方向投递可以有多份附件，但基础事实不能互相冲突。
- BOSS/猎聘等聊天沟通场景先用短文本证明匹配点，不要把整段自我介绍塞给对方。
- 国内求职信不是默认必需；更多时候使用内推私信、BOSS 开场白、邮件正文或投递备注。

## 经历包装与能力转译

合理包装的目标是把真实经历写成目标岗位能识别的证据，不是制造不存在的能力。

输出包装建议时，不要只给用户一句“可以写成 X”。要让用户看懂包装逻辑，知道为什么这样写、哪里不能越界、补什么证据后能写得更强。

常用方法：

```markdown
| 方法 | 适用场景 | 写法 |
| --- | --- | --- |
| 场景重命名 | 原经历名称不专业，但任务有价值 | 把“帮社团做活动”转成“活动执行与现场协调” |
| 能力转译 | 经历和岗位不完全同名，但能力相通 | 把“给同学讲题”转成“教学表达、用户理解与反馈调整” |
| 结果强调 | 做过事情但表达太平 | 补充服务对象、数量、交付物、反馈或复盘 |
| 顺序重排 | 同一素材适配不同方向 | 投运营时先写用户/内容/增长，投产品时先写需求/流程/反馈 |
| 短期补位 | 技能暂时不足，但可快速补材料 | 做 1 页案例复盘、作品截图说明、竞品分析或 Demo 录屏 |
```

轻量包装卡：

```markdown
## 包装卡

- 原始经历：
- 目标岗位要识别的能力：
- 可包装方向：
- 简历可写：
- 面试可说：
- 不能写：
- 补完证据后可增强为：
```

同一段经历可给两档表达：

```text
当前可用版：基于已有事实和证据，克制表达。
证据增强版：补截图、链接、录屏、复盘、数字或第三方反馈后再使用。
```

传统行业到新岗位的转译示例：

```markdown
| 原经历 | 可转译能力 | 写法方向 |
| --- | --- | --- |
| 工厂生产/班组管理 | 流程管理、排期、质量控制 | 生产计划、异常处理、跨班组协同 |
| 门店销售 | 客户沟通、转化、需求判断 | 客户分层、销售转化、复购维护 |
| 行政/人事 | 组织运营、流程规范、活动执行 | 制度落地、员工沟通、供应商协调 |
| 客服/售后 | 用户体验、问题定位、情绪安抚 | 工单分析、用户反馈、服务流程优化 |
| 教育/培训 | 课程设计、表达、反馈迭代 | 培训赋能、学习路径、效果评估 |
```

表达强度要匹配真实角色：

```text
主导/负责：候选人拥有关键决策或主要交付责任。
参与/协助：候选人完成明确模块，但不是负责人。
支持/配合：候选人贡献较小或执行局部任务。
学习/实践：候选人能解释方法，但缺少真实业务交付。
```

如果候选人马上要投递，优先做：

```text
已有经历转译
关键证据补图/补链接/补说明
成果数字补采
1-2 个短案例页
面试解释口径
不能写的高风险表述清单
```

## 求职信 / 私信 / 邮件 / 平台开场白

只有在能补充简历看不出的上下文时使用：

```text
为什么是这个岗位
为什么是这个候选人
1-2 个证据点
可沟通时间 / 下一步
```

保持具体、简短，不写泛泛模板话。

## 简历视觉最低建议

- 白底黑字或低饱和配色，单栏优先。
- 中文字体用微软雅黑、黑体、宋体等常见字体。
- 字号 10.5-12pt，行距 1.2-1.5，留足页边距。
- 不用花哨模板、复杂图形、技能条和扫描图片。
- 中英双语简历要保证时间线、公司、岗位、项目名完全一致。
