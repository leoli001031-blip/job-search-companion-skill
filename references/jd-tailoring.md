# JD 拆解、匹配判断与岗位定制

用于目标岗位样本拆解、具体 JD 匹配判断和投递前材料定制。

## 两种使用时机

1. **目标岗位 / JD 样本拆解：**基础素材集合之后，根据用户想投的岗位、行业、城市、阶段和限制条件，拆少量样本形成岗位要求地图，用来指导证据补挖、技能分层和求职叙事。
2. **具体 JD 定制：**拿到某个岗位后，结合公司调研逐条匹配，判断投递优先级，再定制简历、私信、求职信和面试准备。

不要把 JD 分析只放在材料输出前，也不要在素材整理前过早裁剪候选人经历。基础素材集合回答“这个人有什么”，用户目标回答“现在服务哪个方向”，岗位样本回答“这个方向通常要什么”，具体 JD 和公司背景回答“这次怎么投、怎么讲”。

不要替用户断定“适合做什么工作”。如果用户目标模糊，可以帮助澄清需求、列出选项、说明匹配风险和准备成本；如果用户目标明确，直接围绕目标岗位推进。

```markdown
- 常见交付物：
- 协作对象：
- 结果指标：
- 硬性门槛：
- 能力簇：

## 素材动作
- 已有强证据：
- 需要补挖的经历：
- 需要寻找的证据：

```

```markdown
| --- | --- | --- | --- | --- |
```

## 具体 JD 拆解

拿到具体岗位后先拆解，再定制材料。不要只提取关键词。

```markdown
# JD 拆解

## 岗位画像
- 岗位名称：
- 行业/业务线：
- 工作对象：
- 阶段层级：实习/校招/初级/中级/高级/管理/项目制

## 工作任务
- 日常高频任务：
- 关键交付物：
- 协作对象：
- 结果指标：

## 门槛与能力
- 硬性门槛：学历/专业/年限/证书/语言/地点/到岗/排班
- 必需经验或工具：
- 岗位核心能力：
- 行业/业务理解：
- 沟通协作：
- 执行与结果：
- 学习潜力或稳定性信号：

## 信号
- 明确加分项：
- 必须命中的关键词：
- 不应硬塞的关键词：
- 公司真正可能在筛什么：
- 面试官可能担心什么：
- 岗位可能存在的风险：
```

具体公司研究见 `references/company-research.md`。

## JD 到候选人的映射

拆完 JD 和公司背景后，再映射候选人证据：

```markdown
| JD / 公司要求 | 候选人证据 | 证据强度 | 材料动作 | 风险 |
| --- | --- | --- | --- | --- |
| 硬性门槛 |  | 强/中/弱/缺口 | 强调/解释/暂缓/跳过 |  |
| 高频任务 |  | 强/中/弱/缺口 | 强调/重写/补案例/追问 |  |
| 关键交付物 |  | 强/中/弱/缺口 | 放简历/作品集/面试准备 |  |
| 能力簇 |  | 强/中/弱/缺口 | 技能区/经历要点/个人总结 |  |
| 隐含信号 |  | 强/中/弱/缺口 | 预设解释/STAR 故事 |  |
| 公司业务问题 |  | 强/中/弱/缺口 | 私信/面试表达/反问 |  |
```

材料动作规则：

- 重新排序和重写已有证据。
- 使用目标岗位语言，但不能加入不受支持的技能。
- 必需关键词没有证据时列为 gap，不要硬塞。
- JD 写得很虚时，按行业、层级、工作对象和交付物推断岗位本质，并标明“这是推断”。
- 转型或 0 经验候选人优先连接可迁移结果、项目、作品和能被追问的故事。

## 薄弱点补充挖掘

具体 JD 暴露薄弱点时，先查已有素材集合和经历卡片；确实缺证据时再补问。

```markdown
| 薄弱点 | 可能证明方式 | 引导问题 |
| --- | --- | --- |
| 沟通协作 | 跨团队协调、客户/用户反馈、会议纪要、活动记录 | 你有没有协调不同角色完成一件事？分歧是什么，怎么推进，有没有反馈或结果截图？ |
| 执行力 | 按期交付、处理量、流程清单、复盘文档 | 有没有一件事是你从头跟到尾落地的？周期多长，交付了什么？ |
| 数据/结果 | 人数、收入、转化、满意度、效率、排名 | 这件事影响了多少人、多少钱、多少次、多久？没有数字时有没有交付物或第三方反馈？ |
| 业务理解 | 调研报告、竞品分析、课程作业、访谈、复盘文章 | 你有没有研究过这个行业、用户或业务？有没有报告、笔记或可展示判断？ |
| 作品/项目 | 截图、Demo、PPT、文档、视频、公开链接、证书 | 有没有任何能展示过程或结果的文件？草稿、录屏、汇报材料也可以。 |
```

补挖结果标记为：可直接写入简历、适合放作品集/案例页、只适合面试解释、仍缺证据不能硬写。

## 短期补位与合理包装

马上要投递时，先判断缺口类型：

```markdown
| 缺口类型 | 处理策略 | 边界 |
| --- | --- | --- |
| 表述没贴合 | 用 JD 语言重写真实经历 | 不增加没做过的任务 |
| 能力相通但场景不同 | 做能力转译，说明迁移逻辑 | 不伪装成同岗位经验 |
| 有经验但缺证据 | 补截图、链接、案例页、复盘文档 | 不写无法确认的数字 |
| 硬技能了解但不熟 | 写“了解/实践过/正在补强” | 不写“熟练掌握/独立负责” |
| 核心门槛完全缺失 | 暂缓、带解释投递或跳过 | 不硬塞关键词 |
```

每个包装点同时给三种口径：简历可写版本、面试解释版本、不能写版本。

## JD 红旗与真实性判断

这些词不一定代表坏岗位，但需要追问确认：

```markdown
| JD 表述 | 可能含义 | 追问方式 |
| --- | --- | --- |
| 弹性工作 | 可能加班或边界不清 | 日常工作时间和加班频率大概怎样？ |
| 抗压能力强 | 目标、客户或人手压力 | 压力主要来自交付周期、客户还是内部协作？ |
| 综合薪资 X-Y | 底薪可能低，浮动占比高 | 固定 base、绩效和补贴分别是多少？ |
| 高强度/有激情 | 可能长期加班或资源不足 | 高强度主要发生在什么阶段，是否常态化？ |
| 扁平化管理 | 可能一人多岗、流程不成熟 | 直属上级、协作团队和岗位优先级是什么？ |
```

长期挂着的 JD 可能是人才池、市场摸底或岗位未批。可问：

```text
这个岗位目前是新增 HC、替补岗位，还是长期人才储备？
现在流程推进到什么阶段？业务部门是否已经明确要人？
```

薪资范围可能只是平台展示区间。涉及“综合薪资”“最高可达”“面议”时，必须确认固定和浮动部分。

## 投递判断与输出

定制材料前先判断：立即投递、带解释投递、暂缓、跳过。

- **立即投递：**必需项有强证据，岗位匹配清楚，没有关键硬伤。
- **带解释投递：**有缺口，但可迁移经历或项目证据能可信补位。
- **暂缓：**有机会，但缺作品集证据、薪资/地点不明，或需要用户确认。
- **跳过：**关键要求没有支撑，岗位明显不匹配，或风险大于收益。

每个具体 JD 至少输出：

```text
公司背景摘要
JD 拆解摘要
匹配矩阵
投递判断
简历策略和 section 顺序
需要强调的 Skills
要使用的项目/经历要点
需要补挖的薄弱点
面试中的公司理解口径
可以问面试官的问题
不能写的表述
```
