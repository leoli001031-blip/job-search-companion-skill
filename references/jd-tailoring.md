# JD 拆解、匹配判断与岗位定制

用于目标岗位样本拆解、具体 JD 匹配判断和投递前材料定制。

## 两种使用时机

1. **目标岗位 / JD 样本拆解：**基础素材集合之后，根据用户想投的岗位、行业、城市、阶段和限制条件，拆少量样本形成岗位要求地图，用来指导证据补挖、技能分层和求职叙事。
2. **具体 JD 定制：**拿到某个岗位后，结合公司调研逐条匹配，判断投递优先级，再定制简历、私信、求职信和面试准备。

不要把 JD 分析只放在材料输出前，也不要在素材整理前过早裁剪候选人经历。基础素材集合回答“这个人有什么”，用户目标回答“现在服务哪个方向”，岗位样本回答“这个方向通常要什么”，具体 JD 和公司背景回答“这次怎么投、怎么讲”。

不要替用户断定“适合做什么工作”。如果用户目标模糊，可以帮助澄清需求、列出选项、说明匹配风险和准备成本；如果用户目标明确，直接围绕目标岗位推进。

## 具体 JD 拆解

拿到具体岗位后先拆解，再定制材料。不要只提取关键词。

### 具体 JD 双输入流程

具体 JD 定制必须同时使用两个输入：

```text
JD 原文：岗位职责、要求、硬门槛和关键词
公司信息：业务类型、产品/服务、发展阶段、岗位上下文和公开动态
```

如果 JD 包含公司名、品牌名或可识别业务线，先按 JD 原文拆出岗位画像，再立即读取 `references/company-research.md` 做公司理解卡；完成后再进入候选人匹配。不要在只看 JD 的情况下直接写简历、投递话术或面试口径。

如果没有公司名，输出“公司未知 / 未提供”，只基于 JD 和行业共性判断，并提示拿到公司信息后需要重做公司理解与岗位真实意图判断。

```markdown
- 工作对象：
- 阶段层级：实习/校招/初级/中级/高级/管理/项目制

- 必需经验或工具：
- 岗位核心能力：
- 行业/业务理解：
- 沟通协作：
- 执行与结果：
- 学习潜力或稳定性信号：

## 信号
- 明确加分项：
- 必须命中的关键词：
- 不应硬塞的关键词：
- 公司真正可能在筛什么：

```

具体公司研究见 `references/company-research.md`，输出优先使用“公司理解卡”而不是完整长报告。

## JD 题目化与逐题作答

把 JD 当作题目，不把简历当成自我介绍。每个核心要求都要判断候选人如何作答，不能让 HR 或业务面试官自己联想。

输出具体 JD 匹配时，先把 JD 拆成 5-8 道关键题：

```markdown
| JD 题目 | 面试官真正想确认什么 | 当前答案 | 答题载体 | 缺口动作 |
| --- | --- | --- | --- | --- |
| [要求 1] | [能力/经验/交付物/风险] | 已做过/可迁移/可短期证明/缺失 | 简历/作品集/私信/面试故事/学习附录 | 重写/补挖/做作品/补学习/暂缓/跳过 |
```

答题规则：

- **已做过：**用简历 bullet 和面试故事直接回答，必要时补截图、链接或数字。
- **能力相通：**用可迁移经历回答，说明相同底层能力和不同场景边界，不伪装成同岗位经验。
- **可短期证明：**不把“学习中”写成“熟练”，但可以用调研、案例页、流程图、Demo、模板、复盘或小样本测试证明理解和执行潜力；同时输出学习附录，列出需要补的概念、工具、练习任务和完成标准。
- **完全缺失：**不硬塞关键词；判断暂缓、带解释投递或跳过，并给出学习附录。学习附录可以“编”练习题、模拟任务、作品方案和补课路径，但不能编造成候选人已经完成的经历或成果。

一份材料不必回答所有题：简历回答最重要的 3-5 题，作品集回答证据复杂的题，投递话术回答最强钩子，面试故事回答需要展开的题。

### 岗位痛点与主动提案

JD 不只是字面要求，也是在表达招聘方想减轻的某种压力：没人推进、没人整理、没人交付、没人把复杂事讲清楚、没人把工具真正用起来。不要机械背 JD，也不要让招聘方替候选人想“你能帮我做什么”。

对高价值岗位，输出一张主动提案表：

```markdown
| JD/公司信号 | 招聘方可能的痛点 | 候选人可以怎么帮 | 可展示证据 | 需要验证的问题 |
| --- | --- | --- | --- | --- |
| [职责/业务/阶段] | [效率/交付/增长/协作/采用/质量] | [2-3 个具体动作] | [简历/作品/案例/模拟材料] | [面试或内推时确认] |
```

主动提案规则：

- 不说“我什么都可以做”，要说“我可以先帮你做 A/B/C”。
- 不把公司问题说成内部事实；用“我基于公开信息推测 / 我想进一步确认”。
- 提案要小而具体：一页调研、一个流程模板、一版 Demo、一个培训 FAQ、一张指标看板，比空泛热情更有说服力。
- 对外表达要尊重原团队，不用“你们这里做得不好”这类攻击性说法；改成“我基于公开信息做了一版初步思考，想请教是否接近真实场景”。

## JD 到候选人的映射

拆完 JD 和公司背景后，再映射候选人证据：

```markdown
| JD / 公司要求 | 候选人证据 | 证据强度 | 答题状态 | 材料动作 | 风险 |
| --- | --- | --- | --- | --- | --- |
| 硬性门槛 |  | 强/中/弱/缺口 | 已答/弱答/未答 | 强调/解释/暂缓/跳过 |  |
| 高频任务 |  | 强/中/弱/缺口 | 已答/弱答/未答 | 强调/重写/补案例/追问 |  |
| 关键交付物 |  | 强/中/弱/缺口 | 已答/弱答/未答 | 放简历/作品集/面试准备 |  |
| 能力簇 |  | 强/中/弱/缺口 | 已答/弱答/未答 | 技能区/经历要点/个人总结 |  |
| 隐含信号 |  | 强/中/弱/缺口 | 已答/弱答/未答 | 预设解释/STAR 故事 |  |
| 公司业务问题 |  | 强/中/弱/缺口 | 已答/弱答/未答 | 私信/面试表达/反问 |  |
```

材料动作规则：

- 重新排序和重写已有证据。
- 使用目标岗位语言，但不能加入不受支持的技能。
- 必需关键词没有证据时列为 gap，不要硬塞。
- JD 写得很虚时，按行业、层级、工作对象和交付物推断岗位本质，并标明“这是推断”。
- 转型或 0 经验候选人优先连接可迁移结果、项目、作品和能被追问的故事。

## 薄弱点补充挖掘

具体 JD 暴露薄弱点时，先查已有素材集合和经历卡片；确实缺证据时再补问。

```markdown
| 薄弱点 | 可能证明方式 | 引导问题 |
| --- | --- | --- |
| 沟通协作 | 跨团队协调、客户/用户反馈、会议纪要、活动记录 | 你有没有协调不同角色完成一件事？分歧是什么，怎么推进，有没有反馈或结果截图？ |
| 执行力 | 按期交付、处理量、流程清单、复盘文档 | 有没有一件事是你从头跟到尾落地的？周期多长，交付了什么？ |
| 数据/结果 | 人数、收入、转化、满意度、效率、排名 | 这件事影响了多少人、多少钱、多少次、多久？没有数字时有没有交付物或第三方反馈？ |
| 业务理解 | 调研报告、竞品分析、课程作业、访谈、复盘文章 | 你有没有研究过这个行业、用户或业务？有没有报告、笔记或可展示判断？ |
| 作品/项目 | 截图、Demo、PPT、文档、视频、公开链接、证书 | 有没有任何能展示过程或结果的文件？草稿、录屏、汇报材料也可以。 |
```

补挖结果标记为：可直接写入简历、适合放作品集/案例页、只适合面试解释、仍缺证据不能硬写。

## 短期补位与合理包装

马上要投递时，先判断缺口类型：

```markdown
| 缺口类型 | 处理策略 | 边界 |
| --- | --- | --- |
| 表述没贴合 | 用 JD 语言重写真实经历 | 不增加没做过的任务 |
| 能力相通但场景不同 | 做能力转译，说明迁移逻辑 | 不伪装成同岗位经验 |
| 有经验但缺证据 | 补截图、链接、案例页、复盘文档 | 不写无法确认的数字 |
| 硬技能了解但不熟 | 写“了解/实践过/正在补强” | 不写“熟练掌握/独立负责” |
| 核心门槛完全缺失 | 暂缓、带解释投递或跳过 | 不硬塞关键词 |
```

每个包装点同时给三种口径：简历可写版本、面试解释版本、不能写版本。

### JD 缺口到作品证明矩阵

当 JD 的某道题在简历里答不充分，但短期可以证明理解和执行潜力时，优先设计“小而准”的作品，不要求做大型项目。

```markdown
| JD 要求 | 用户当前不会/证据弱时 | 可做的短期作品 | 简历/面试边界 |
| --- | --- | --- | --- |
| 需求调研 / 用户洞察 | 没有正式调研经验 | 访谈提纲、用户画像、痛点地图、需求池 | 写“调研/拆解/整理”，不写“负责用户研究体系” |
| 项目管理 / 推进落地 | 没有正式项目经理经历 | 项目排期表、风险台账、交付清单、周报模板 | 写“设计项目推进模板/参与推进”，不写“主导复杂项目” |
| AI 工具评估 | 没有企业 AI 落地经验 | 工具对比表、小场景测试记录、评测指标表 | 写“评估/试用/对比”，不写“完成企业级评测体系” |
| 产品运营 / 用户成功 | 没有同名岗位经验 | Onboarding 流程、FAQ、培训材料、反馈闭环看板 | 写“设计采用流程/培训材料样例”，不写“负责用户成功” |
| 业务理解 / 行业理解 | 缺少目标行业经历 | 公司理解卡、竞品分析、业务场景拆解报告 | 标注基于公开资料和行业共性，不说成内部事实 |
| 数据分析 / 效果评估 | 没有真实业务数据 | 指标树、样例看板、口径说明、模拟复盘 | 使用模拟/样例限定词，不写真实提升 |
| 内容运营 / 培训赋能 | 缺少内容案例 | 使用指南、案例文章、培训脚本、短视频脚本 | 写交付物和适用场景，不夸大发布效果 |
| 技术协作 / 工程沟通 | 不是技术岗或代码不深 | API 流程图、PRD、Demo README、技术沟通清单 | 写“能与技术沟通需求和边界”，不伪装成工程交付 |
```

作品设计要回答：

```text
这件作品回答了 JD 的哪道题？
它证明的是理解、执行、协作、判断还是真实业绩？
需要放在简历、作品集、投递附件还是面试演示？
哪些表述可以说，哪些必须留到追问时解释？
```

### 学习附录

当用户对 JD 的某些要求暂时不会、只了解概念或没有可证明经历时，输出学习附录。学习附录不是投递版简历正文，而是告诉用户“为了回答这道 JD 题，接下来应该补什么”。

```markdown
## 学习附录：针对 JD 缺口的补强清单

| JD 题目 | 需要补的知识 | 需要补的工具/方法 | 练习任务 | 完成标准 | 完成后可放入哪里 |
| --- | --- | --- | --- | --- | --- |
| [要求 1] | [概念/行业/流程] | [工具/模板/方法] | [模拟任务或小作品] | [可展示产物/能讲清的问题] | 简历/作品集/面试故事/只作内部准备 |
```

学习附录可以生成：

- 概念清单：用户需要理解的术语、流程、指标和常见误区。
- 练习题：围绕目标岗位设计的小任务、模拟场景、案例拆解和复盘问题。
- 小作品方案：调研页、流程图、Demo、评估表、培训材料、FAQ、需求池、指标树、看板样例。
- 面试自测题：用户学完后应该能回答的问题。

### 模拟补足到可写材料

如果用户愿意继续做补足，不要只停在“建议学习”。可以直接带用户完成一轮小模拟：

```text
选择 1 道最关键 JD 题
-> 给出 1 个贴近目标岗位的模拟场景
-> 让用户完成调研、拆解、表格、流程图、Demo、话术或复盘中的一个小产物
-> 帮用户复核产物是否回答了 JD
-> 通过后转成简历 bullet、作品集说明和面试讲法
```

完成后可以写入：

- **简历：**写成“个人项目 / 模拟案例 / 自测 Demo / 作品样例”，强调做了什么、产出了什么、如何验证理解。
- **作品集：**展示输入、分析过程、输出物、复盘和下一步改进。
- **面试：**说明这是为理解岗位场景做的个人练习，重点讲方法、判断和可迁移能力。

边界：

- 不能把学习附录里的练习任务写成候选人已完成的真实工作。
- 用户完成练习并提供材料或明确确认后，才能把它转成“个人项目 / 作品样例 / 面试展示”。
- 如果只是学习计划，简历正文不写；如果已有可展示成果，可在作品集或面试中用“个人练习 / 模拟案例 / 自测 Demo”限定。
- 简历上可以写“基于公开资料完成 X 场景模拟 / 设计 X 模板 / 搭建 X Demo”，不能写“为某公司完成 X 项目 / 帮客户提升 X% / 推动业务上线”。

## JD 红旗与真实性判断

这些词不一定代表坏岗位，但需要追问确认：

```markdown
| JD 表述 | 可能含义 | 追问方式 |
| --- | --- | --- |
| 弹性工作 | 可能加班或边界不清 | 日常工作时间和加班频率大概怎样？ |
| 抗压能力强 | 目标、客户或人手压力 | 压力主要来自交付周期、客户还是内部协作？ |
| 综合薪资 X-Y | 底薪可能低，浮动占比高 | 固定 base、绩效和补贴分别是多少？ |
| 高强度/有激情 | 可能长期加班或资源不足 | 高强度主要发生在什么阶段，是否常态化？ |
| 扁平化管理 | 可能一人多岗、流程不成熟 | 直属上级、协作团队和岗位优先级是什么？ |
```

长期挂着的 JD 可能是人才池、市场摸底或岗位未批。可问：

```text
这个岗位目前是新增 HC、替补岗位，还是长期人才储备？
现在流程推进到什么阶段？业务部门是否已经明确要人？
```

薪资范围可能只是平台展示区间。涉及“综合薪资”“最高可达”“面议”时，必须确认固定和浮动部分。

## 投递判断与输出

定制材料前先判断：立即投递、带解释投递、暂缓、跳过。

- **立即投递：**必需项有强证据，岗位匹配清楚，没有关键硬伤。
- **带解释投递：**有缺口，但可迁移经历或项目证据能可信补位。
- **暂缓：**有机会，但缺作品集证据、薪资/地点不明，或需要用户确认。
- **跳过：**关键要求没有支撑，岗位明显不匹配，或风险大于收益。

每个具体 JD 的后台分析至少覆盖：

```text
公司背景摘要
JD 拆解摘要
匹配矩阵
投递判断
简历策略和 section 顺序
需要强调的 Skills
要使用的项目/经历要点
需要补挖的薄弱点
面试中的公司理解口径
可以问面试官的问题
不能写的表述
```

前台默认不要一次性全部展开。普通对话优先输出：

```markdown
## 当前判断
- 投递建议：立即投递 / 带解释投递 / 暂缓 / 跳过
- 最需要回答的 JD 题：3-5 道
- 当前最强证据：1-3 个
- 当前最大缺口：1-3 个
```
